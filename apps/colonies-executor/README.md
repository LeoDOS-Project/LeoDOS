# cFS Rust Application with Colonies Executor

This directory contains a Rust application for NASA's Core Flight System (cFS) that integrates with the Colonies distributed task execution system.

## Directory Structure

```
apps/rust/
├── CMakeLists.txt          # CMake configuration (integrates Rust into cFS build)
├── config/                 # cFS configuration files
│   ├── default_rust_msgids.h     # Message IDs for Software Bus
│   ├── default_rust_perfids.h    # Performance IDs
│   └── ...
└── fsw/                    # Flight Software (Rust crate)
    ├── Cargo.toml          # Rust project configuration
    ├── build.rs            # Build script (generates cFS bindings)
    ├── src/
    │   ├── lib.rs          # Main application entry point
    │   └── colonies/       # Colonies executor module
    │       ├── mod.rs      # Public API
    │       ├── core.rs     # Data structures (Process, FunctionSpec, etc.)
    │       ├── crypto.rs   # Cryptographic signing (ECDSA)
    │       ├── http.rs     # HTTP client using OSAL sockets
    │       └── rpc.rs      # RPC message builders
    └── target/             # Build output (generated by Cargo)
        ├── i686-unknown-linux-gnu/
        │   └── debug/
        │       ├── build/rust-*/out/bindings.rs  # Generated FFI bindings
        │       └── librust.so                    # Compiled library
        └── ...
```

## How It Works

### CMake Integration

1. **Configuration Phase** (`make prep`):
   - CMake reads `CMakeLists.txt`
   - Sets up paths and environment variables
   - Generates build files

2. **Build Phase** (`make rust`):
   - CMake invokes `cargo build`
   - Cargo compiles the Rust code to a `.so` library
   - Output: `fsw/target/i686-unknown-linux-gnu/debug/librust.so`

3. **Installation Phase** (`make install`):
   - CMake copies `librust.so` to `/exe/cpu1/cf/rust.so`
   - cFS loads this at runtime

### Rust to cFS Bridge

The bridge between Rust and cFS happens through:

1. **cfs_bindings module**: Rust FFI bindings to cFE/OSAL/PSP C APIs
   - Generated at compile time by `build.rs` using bindgen
   - `build.rs` runs before the main code is compiled
   - Generates `bindings.rs` with all C function signatures, structs, and constants
   - Included in `src/cfs_bindings.rs` via `include!(concat!(env!("OUT_DIR"), "/bindings.rs"))`
   - No external dependency required - everything is self-contained

2. **Entry Point**: `Rust_AppInit()` in `lib.rs`
   - `#[no_mangle]`: Tells Rust not to change the function name
   - `pub extern "C"`: Makes it callable from C with C calling convention
   - cFS calls this function to start the app

3. **Build Configuration**: `Cargo.toml`
   - `crate-type = ["cdylib"]`: Build as a C-compatible dynamic library
   - Target: `i686-unknown-linux-gnu` (32-bit x86 Linux)

## Colonies Executor

The app implements a Colonies executor that:

1. **Polls for work**: Calls `colonies::assign()` to get tasks from Colonies server
2. **Executes tasks**: Runs the assigned function (currently supports "say")
3. **Reports results**: Calls `colonies::close()` or `colonies::fail()`
4. **Sends telemetry**: Updates counters in cFS telemetry packets

### Architecture

- **OSAL sockets**: Uses cFS `OS_SocketOpen()`, `OS_SocketConnect()`, etc.
- **Manual HTTP**: Builds HTTP/1.1 requests and parses responses
- **#[no_std]**: Uses `alloc` for heap, but no standard library
- **Crypto**: ECDSA signing using pure Rust `k256` library

### Configuration

Edit `src/lib.rs` to configure:

```rust
const COLONIES_COLONY_NAME: &str = "dev";
const COLONIES_EXECUTOR_PRVKEY: &str = "ddf7f...";
const COLONIES_ASSIGN_TIMEOUT: i32 = 10;
```

Server host/port in `src/colonies/http.rs`:

```rust
const COLONIES_SERVER_HOST: &str = "localhost";
const COLONIES_SERVER_PORT: u16 = 50080;
```

## Building

### Prerequisites

- Rust toolchain with i686-unknown-linux-gnu target
- CMake 3.x
- cFS build environment

### Build Commands

```bash
# In Docker container (recommended):
docker run --rm -v $(pwd):/cfs -w /cfs/build/native/default_cpu1 cfs-build make rust

# Outside Docker (if you have the right environment):
cd build/native/default_cpu1
make rust
```

### Build Modes

- **Debug** (default): Includes debug symbols, larger binary (~8.6 MB)
- **Release**: Optimized, smaller binary (~2-3 MB)

To build in release mode, edit `CMakeLists.txt`:

```cmake
set(RUST_TARGET_DIR "${CMAKE_CURRENT_SOURCE_DIR}/fsw/target/i686-unknown-linux-gnu/release")
# And change:
cargo build --release --target=...
```

## Dependencies

### Rust Crates (from Cargo.toml)

- **serde**: Serialization framework
- **serde_json**: JSON encoding/decoding
- **base64**: Base64 encoding (for RPC messages)
- **embedded-alloc**: Heap allocator for no_std
- **k256**: Elliptic curve cryptography (secp256k1)
- **sha2**: SHA-256 hashing
- **hex**: Hexadecimal encoding

All crates are configured with `default-features = false` for no_std compatibility.

## Troubleshooting

### "cannot find -lcryptolib"

This error means the external crypto library isn't available. The current implementation uses pure Rust crypto (k256), so this shouldn't occur. If it does, check that `crypto.rs` doesn't reference `#[link(name = "cryptolib")]`.

### "Unable to generate bindings" from build.rs

This means bindgen couldn't find the C headers. Check:
1. Environment variables are set correctly: `CFE_DIR`, `OSAL_DIR`, `PSP_DIR`
2. The headers exist at those paths
3. clang is installed (bindgen needs it)

### Binary too large

- Use `--release` mode instead of `--debug`
- Strip symbols: `strip librust.so`
- Enable LTO in Cargo.toml (already enabled in release profile)

### cFE API mismatch

If you see undefined references to cFE functions, the bindings may be stale:
```bash
cargo clean  # Clear build cache
make rust    # Rebuild (this regenerates bindings.rs)
```

## Testing

To test the Colonies executor:

1. **Start Colonies server**:
   ```bash
   colonies dev
   ```

2. **Run cFS with the Rust app**:
   ```bash
   cd build/exe/cpu1
   ./core-cpu1
   ```

3. **Submit a task**:
   ```bash
   colonies function exec --func say --args "hello world" --targettype cli --wait
   ```

4. **Check output**: The executor should pick up the task and echo "hello world"
